WebCodecs 브라우저 프록시 렌더링 — 설계

"올리는 사람 컴퓨터로 인코딩, 설치 0, 많은 사용자"를 만족하는 구조. 비싼 재인코딩만 브라우저(WebCodecs 하드웨어 가속)로 옮기고, 서버는 싼 remux·서빙·폴백만. 현재 서버가 쓰는 h264_videotoolbox 인코딩을 사용자 GPU로 분산하는 셈.

1현재 vs WebCodecs

2브라우저 파이프라인 (WebCodecs)

파일 선택 → 디먹스 → 디코드 → 다운스케일 → 인코드 → mux → 업로드. 디코드·인코드가 하드웨어 가속(WASM 아님).
1
디먹스 mp4box.js
업로드 파일(mp4/mov)을 스트리밍으로 파싱 → 인코딩된 비디오/오디오 청크. 파일 통째로 메모리에 안 올림.
2
디코드 VideoDecoder
WebCodecs로 프레임 복원.
HW 가속
3
다운스케일 OffscreenCanvas / VideoFrame
프록시 해상도로 축소(예: 720p). 검수용이라 원본 해상도 불필요.
4
인코드 VideoEncoder (H.264)
표준 프로필로 재인코딩(예: 720p·~3Mbps·H.264). 기기 GPU의 VideoToolbox/NVENC 사용.
HW 가속
5
오디오 AudioEncoder (AAC)
AAC로 인코드(원본이 AAC면 copy).
6
mux mp4box.js → 작은 MP4(faststart)
단일 프록시 MP4 생성. HLS 조각은 서버가 싸게 함(브라우저 복잡도↓).
7
업로드 — 프록시 먼저 / 원본 백그라운드
프록시(작음)를 먼저 올려 검수 즉시 가능, 원본(납품·아카이브)은 뒤에 천천히. 진행률 UI는 프레임 기준.

3서버 변경 (작음) + 폴백

프록시 수신 → remuxffmpeg -i proxy.mp4 -c copy -f hls … — 재인코딩 없음. 기존 fingerprint·HLS 저장·/api/stream·HlsVideo 그대로 재사용. 재활용
검증프록시의 길이·해상도·코덱이 원본과 대략 맞는지 가벼운 sanity check(내부툴이라 가볍게). 원본 = source of truth.
폴백브라우저가 WebCodecs/H.264 인코더 미지원이거나 실패 → 클라가 원본만 업로드 → 서버가 기존 h264_videotoolbox 인코딩(현재 경로 그대로). 코드 손 안 댐. 안전망
프로필 고정720p·H.264 main·~3Mbps·AAC 등 한 프로필로 표준화 → 기기 달라도 일관된 프록시.

4핵심 결정

질문답
프록시 포맷단일 MP4(faststart) 추천 — 브라우저에서 HLS까지 만들면 복잡. 서버가 -c copy로 HLS 조각(거의 무료). 적응형 비트레이트 필요해지면 그때 다중 렌디션.
원본은?여전히 업로드(납품·아카이브) — 단 프록시 먼저, 원본 백그라운드. 검수는 프록시로 즉시.
안 되는 브라우저서버 폴백(현재 경로). 커버리지 100% 유지.
비용인코딩 $0(사용자 GPU). Mux·Cloudflare Stream 같은 분당 과금 회피 — 사람 많을수록 차이 큼.

5PoC 계획 (먼저 검증할 것)

전면 통합 전에 독립 페이지로 핵심 위험부터 측정. "되는지/빠른지/지원되는지"를 먼저.
1
독립 PoC 페이지 — 파일 선택 → WebCodecs 디코드→다운스케일→인코드→mux MP4 → 다운로드. (업로드·통합 없이 인코딩만)
2
측정 — 핵심 지표로 갈지말지 판단
인코딩 속도 ×실시간(예: 5분 영상 1분 내?)출력 크기화질(검수 충분?)메모리(2GB도 OK?)브라우저 커버리지
3
통합 — 좋으면 업로드 플로우에 끼움: 능력 감지 → WebCodecs 인코딩 → 프록시 먼저 업로드 → 서버 remux. 폴백 경로 유지.
4
서버 remux — generateHLS에 "이미 인코딩된 입력은 -c copy" 분기 추가(작음).

6리스크 & 대응

브라우저 지원 — Chrome·Edge·Safari 16.4+는 OK, Firefox 최근/부분, 구형 X. → 능력 감지 후 서버 폴백.
H.264 인코더 가용성 — WebCodecs는 있어도 특정 코덱 인코더는 기기마다 다를 수 있음. → VideoEncoder.isConfigSupported()로 사전 확인, 안 되면 폴백.
구현 난이도 — 디먹스+mux(mp4box.js)에 WebCodecs config(avcC) 연결이 손이 감. → PoC로 먼저 검증, 데모·레퍼런스 존재.
편집 중 경합 — 업로더가 편집 중이면 GPU 경합. → 업로드=대기시간이라 대체로 OK, HW 인코딩은 가벼움. 우선순위 낮게.
모바일/저사양 업로더 — 느릴 수 있음. → 폴백 또는 능력 감지로 서버 처리.
한 줄 — 비싼 재인코딩만 브라우저 WebCodecs(HW)로 → 사용자 GPU에 분산(설치 0·비용 0·스케일), 서버는 remux/서빙/폴백만. 현재 h264_videotoolbox·HLS·/api/stream 인프라 대부분 재사용. 다음 = PoC 페이지로 속도·화질·커버리지 측정 후 갈지 결정.
